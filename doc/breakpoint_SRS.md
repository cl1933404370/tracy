## 1.引言

## 1.1 目的

本软件需求规格说明书（SRS）用于定义 **HCCL/HCOMM 代码 Trace 性能分析工具** 的功能需求、非功能需求、接口规范和使用场景。  
目标读者包括：

- HCCL/HCOMM 开发者  
- 性能优化工程师  
- 工具链维护者  
- 架构师  

---

## 1.2 范围

本工具用于：

- 在 HCCL/HCOMM 源码中插入轻量级性能打点  
- 采集运行时 Trace 数据  
- 对通信路径、算子调度、消息流进行性能分析  
- 提供可视化界面用于展示关键性能指标  

工具必须：

- 支持离线运行  
- 兼容 C++14  
- 高性能、低开销、减少内存碎片

## 2. 总体描述

## 2.1 产品背景

HCCL/HCOMM 是高性能通信库，涉及大量异步调度、消息传递和设备间通信。  
现有工具无法满足：

- 代码级别的精确性能打点  
- 对通信路径的细粒度分析  
- 离线环境下的性能诊断  
- 低开销、高精度的 Trace 采集  

以及打点的痛点：

- cpu gpu需要的打点函数不一样
- 一般stdchrono比直接调用汇编 c函数慢
- 需要手动打点 机械工作 可以ai粗略分析，打点关键地方初步分子然后细化
- 可视化差，无法类似火焰图直接打印图出来
- 频繁得编译代码
- 时间对象创建销毁也耗时间很慢
- 日志也是个问题
- 有tracy可以做到大部分事情，我得学习一下取其精华
- 避免宏
- 保证so间或so内顺序

因此需要一个专用的 **代码级 Trace 工具**。

---

## 2.2 产品目标

本工具旨在：

- 为 HCCL/HCOMM 提供轻量级、可嵌入的性能打点能力  
- 生成可离线分析的 Trace 数据  
- 提供可视化界面用于分析瓶颈  
- 支持开发者快速定位性能问题  

---

|开发者|在代码中插入打点、采集 Trace、分析性能瓶颈|

## 3. 功能需求

## 3.1 代码内性能打点

### REQ-001：支持轻量级打点 API

工具必须提供 C++ API，例如：

```cpp
TRACE_SCOPE("HCCL_Send");
```

### REQ-002：打点开销必须极低

- 单次打点开销 < 100ns  
- 不得引入锁竞争  
- 不得造成内存碎片  

### REQ-003：支持嵌套事件

允许：

- Span 内嵌 Span  
- 自动记录开始/结束时间  

---

## 3.2 Trace 数据采集

### REQ-010：支持事件序列记录

采集内容包括：

- 时间戳  
- 线程 ID  
- 事件类型  
- 标签  
- 持续时间  

### REQ-011：支持多线程采集

不得阻塞业务线程。

### REQ-012：支持离线存储

Trace 文件必须可在无网络环境下分析。

---

## 3.3 性能分析

### REQ-020：支持关键路径分析

识别：

- 最长耗时路径  
- 通信瓶颈  
- 调度延迟  

### REQ-021：支持事件统计

包括：

- 平均耗时  
- P50/P90/P99  
- 调用频率  

---

## 3.4 数据可视化

### REQ-030：支持时间线视图

类似 Perfetto/Tracy：

- 线程时间线  
- Span 展开  
- 事件颜色区分  

### REQ-031：支持统计图表

- 柱状图  
- 饼图  
- 热力图  

---

## 4. 非功能需求

## 4.1 性能需求

- 打点开销 < 100ns  
- 内存碎片率 < 1%  
- Trace 文件写入不阻塞主线程  

---

## 4.2 兼容性需求

- 必须兼容 C++14  
- 支持 Linux（优先）  
- 支持 ARM / x86 架构  

---

## 4.3 可用性需求

- CLI 使用简单  
- 可视化界面无需网络  
- 文档完整  

---

## 4.4 安全性需求

- Trace 文件不包含敏感信息  
- 不上传任何数据（离线）  

---

## 5. 外部接口需求

## 5.1 命令行接口（CLI）

示例：

```bash
trace_tool analyze trace.json --output report.html
```

## 5.2 配置文件接口

支持 YAML/JSON：

```yaml
sampling: 1
buffer_size: 64MB
```

## 5.3 输出数据格式

- JSON
- 二进制（高性能）
- HTML（可视化报告）

## 6. 用例

## 6.1 UC-01：插入性能打点

参与者： 开发者
触发条件： 开发者需要分析某段代码性能
流程：

- 在代码中插入 TRACE_SCOPE
- 编译运行
- 生成 trace 文件

## 6.2 UC-02：生成 Trace 数据

- 程序运行
- 工具自动记录事件
- 写入 trace 文件

## 6.3 UC-03：分析性能瓶颈

- 开发者运行 CLI
- 工具生成统计报告
- 开发者查看瓶颈

## 6.4 UC-04：可视化查看

- 打开 HTML 报告
- 查看时间线
- 查看关键路径
